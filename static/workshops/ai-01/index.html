一个好的 prompt 是什么样的

问题的核心是：
在一个会自动补全、自动收敛、自动替你定义任务的系统里，怎样重新夺回问题定义权。

有些极简单、低风险、一次性的任务，根本不需要复杂 prompt。比如“把这段英文翻成中文”“列 10 个标题备选”。这种场景里，prompt 简短反而更好。因为任务空间本来就小，过度约束会让输出发僵，甚至增加噪音。

一种 prompt 只是触发器。
比如“帮我总结这篇文章”。这种 prompt 能启动任务，但几乎没有控制力。模型会自己补全场景，自己猜重点，自己决定输出方式。很多时候看起来流畅，但其实问题定义权已经让出去了。

另一种 prompt 是约束系统。
它不只说“做什么”，还规定“不要做什么”“遇到不确定怎么办”“输出按什么结构组织”“哪些判断必须标注成推断”。这种 prompt 不是为了让模型更会说，而是为了让它少乱说。

好的 Prompt接近一种最小可运行的任务协议。
它能把四件事同时说清楚：现在要做什么、凭什么做、做到什么算好、做不到时怎么停。很多时候说清楚这些需要大量的文本。

坏 prompt 有时候第一次看起来很惊艳，但一改就塌，因为底层任务定义不稳。

排序 ：同样进入窗口，先后顺序怎么摆

实际排序大致长这样：
系统/企业约束 → 用户或团队级规则 → 项目规则 → 当前任务说明 → 用户显式 pin / @mention 的工作集 → 检索回来的代码片段/文档 → 最近几轮轨迹 → 压缩摘要或记忆条目。
原因不复杂。越往前的内容，越像“约束”；越往后的内容，越像“证据”或“局部状态”。如果把证据排在约束前面，模型就更容易被局部材料带偏。

连续任务

更接近：

真正的失效点，不在 token 不够，而在边界错位
失效一：把规范和轨迹混成一层。
结果是每次摘要都会顺手改写规范，慢慢把“必须遵守”压成“可讨论背景”。
失效二：把继承默认开太大。
自动 memories、宽泛 knowledge、过胖 rules、永不清空的 pinned context，都会让旧任务幽灵一样附在新任务上。公开文档里，各家都在补 memory、rules、knowledge、pinning，但这恰恰意味着污染风险是真实存在的。
失效三：把检索当理解。
Cursor、Windsurf 都在强化索引与搜索，但“能找到相关片段”不等于“形成了正确的当前工作集”。检索强，只是减少盲找，不自动形成判断。
失效四：把缓存当记忆。
prompt caching 优化的是重复前缀的成本和延迟，不负责区分“该留下什么”。这一步跳快了，就会把速度误当成持续智能
失效五：没有切任务协议。
OpenAI 之所以把 compaction 做成原语，也是承认长轨迹直接拖着走会坏。

Memory.md
OpenClaw 的记忆就是写进 workspace 的纯 Markdown，没有隐藏状态。
长期记忆是 MEMORY.md，日记式的短期连续性是 memory/YYYY-MM-DD.md，另外还有实验性的 DREAMS.md。
它还提供 memory_search、memory_get，并在 compaction 前自动做一次 silent memory flush。

官方 issue 里已经有人指出 MEMORY.md 会整份进入 system prompt，造成 token 浪费；也有人指出它会在接近 20 K 字符时被静默截断；还有人提出需要 STATE.md、WORKING.md、VALUE.md 这类额外文件，来承接那些既不像长期记忆、又不该塞回 AGENTS 或 SOUL 的内容。这个现象说明：OpenClaw 已经意识到单纯的 long-term memory 不够了，但更细的分层还在长。

在 Openclaw 里一条消息发出，发生了什么？

Openclaw的上下文引擎是如何取舍如何运作的？

**在 OpenClaw 里，一条消息不是“喂给模型一句话然后等回答”。
它更像是：触发一次绑定在某个 session 上的 agent loop。这个 loop 会先决定这条消息属于谁、落到哪个会话、带上哪些上下文、是否调用工具、结果怎么写回、哪些内容要留到下次。**

OpenClaw 官方把这条主链路直接定义为：接收 → 上下文组装 → 模型推理 → 工具执行 → 流式回复 → 持久化。并且它强调，这是一条“authoritative path”，也就是系统真正把消息变成动作和答复的主路径。

第一步，不是理解消息，而是先给消息找“归属”。
OpenClaw 会先根据来源把消息路由到一个 session。DM 默认共享会话，群聊按群隔离，房间/频道按 room 隔离，cron 每次新会话，webhook 也隔离。对 channel plugin 来说，平台侧的 conversation id、thread id、parent fallback 这些映射，不是模型临场猜出来的，而是插件负责解析，OpenClaw 核心只保留统一的 message 工具和外层 dispatch。也就是说，消息先被变成“某个会话里的一个事件”，不是先变成“一个问题”。

第二步，系统会先处理并发和秩序，而不是立刻思考。
OpenClaw 的 agent loop 会按 session 串行化运行，同一个 session 同时只允许一条 run 真正落地，这样可以避免工具调用和会话写入互相踩踏。若当前已有运行中的任务，新消息会按 queue mode 处理，比如 steer、followup、collect。默认是 collect，也就是把一批消息并成一个后续回合，而不是每来一条都立刻生成一次。这里已经能看出一点原理：OpenClaw 首先是一个会话调度系统，然后才是推理系统。

第三步，才轮到 session 和 workspace 的准备。
在真正送给模型前，OpenClaw 会解析 workspace，必要时创建它；若启用了 sandbox，某些运行还会被重定向到沙盒 workspace。随后它会加载 skills snapshot，解析 bootstrap/context files，并在流式输出前拿到 session write lock，打开 SessionManager。session transcript 默认存在 ~/.openclaw/agents/<agentId>/sessions/<SessionId>.jsonl，而 workspace 则是另一个东西，默认在 ~/.openclaw/workspace。这两个地方不要混淆：session 更像对话历史与运行记录，workspace 更像 agent 的长期操作环境与可注入上下文。

第四步，真正关键的地方来了：上下文装配。
OpenClaw 不是只把聊天历史塞进模型。它会构建 system prompt，把 base prompt、skills prompt、bootstrap context、per-run overrides 组合起来，并且受模型 token 限制和 compaction reserve 约束。文档里还明确说，workspace 里的 AGENTS.md、SOUL.md、TOOLS.md、IDENTITY.md、USER.md、HEARTBEAT.md、BOOTSTRAP.md，以及存在时的 MEMORY.md，都会作为 Project Context 注入；大文件会被裁切并打上标记。这里最重要的一层理解是：OpenClaw 的“人格”“规则”“偏好”“工具习惯”不是散落在聊天里，而是被文件化后反复注入。

第五步，skills 不是装饰，而是“让模型会用工具”的行为说明。
官方把 OpenClaw 分成三层：tools 是 agent 真正调用的函数，skills 是告诉 agent 何时以及如何调用这些工具的 markdown 指南，plugins 则把 channel、tool、skill、model provider 等能力打成包。对模型来说，工具以结构化函数定义的形式出现；对行为来说，skills 以 prompt 中的约束和操作说明出现。所以 OpenClaw 的“智能感”并不只来自模型本体，而是来自“模型 + 可调用函数 + 可注入操作规范”的组合。

第六步，模型开始运行，但它不是孤立生成文本，而是在 loop 里边想边调用。
Agent loop 文档写得很直接：它会建立 pi session，订阅事件流，把 assistant delta、tool event、lifecycle event 桥接回 OpenClaw 的 agent stream。也就是说，你看到的不是一个整块答案突然落下，而是一个带生命周期的流式过程。工具调用前后还可以被 hook 拦截，比如 before_prompt_build、before_agent_reply、before_tool_call、after_tool_call、tool_result_persist。所以一条消息背后，不是一次单纯 completion，而是一条可插桩、可中断、可改写的运行管线。

第七步，记忆不是“模型脑子里还记得”，而是“有没有写回磁盘”。
OpenClaw 的 memory 文档说得非常硬：它通过往 agent workspace 写 Markdown 文件来记忆，没有 hidden state。长期记忆是 MEMORY.md，日记式上下文是 memory/YYYY-MM-DD.md，还有实验性的 DREAMS.md。memory_search 和 memory_get 负责查找和读取这些内容；内建 backend 默认是 SQLite，可做关键词、向量和混合检索。这里真正的结构是：对话历史、被注入的固定上下文、以及被显式写入/检索的记忆，是三套不同东西。

文档对“记忆如何进入当前回合”用了两种表述。 memory overview 说 MEMORY.md 和 daily notes 会自动加载；但 system prompt 文档又强调 daily memory files 不会自动注入，而是通过 memory_search / memory_get 按需访问。更稳妥的理解是：OpenClaw 至少把“长期记忆注入”和“日常记忆检索”分成了两层机制，不要把“会记得”理解成一条单一路径。 这类细节属于高波动实现面，版本演化时最容易变化。

第八步，历史不是无限累加，而是会被压缩。
当会话逼近上下文窗口时，OpenClaw 会 compact older turns，把旧消息总结成 compact entry，写回 transcript，保留近端消息不动。官方还特别强调，完整历史仍然在磁盘上，compaction 改变的是下一轮模型看到什么，不是把历史物理删除。更关键的是，在 compaction 前它会先触发一次 silent turn，提醒 agent 把重要内容写入 memory files，避免总结时丢失关键上下文。也就是说，OpenClaw 的连续性不是靠无限长上下文维持的，而是靠“摘要化 transcript + 文件化记忆”共同维持。

第三种岗位，
是“巡检员”。

这恰好是 OpenClaw 很适合的地方。它有两个容易被混在一起的机制。Heartbeat 是定期在主 session 里唤醒 agent，看有没有需要提醒你的事情；Cron 是 Gateway 内置的调度器，持久化任务、到点唤醒 agent，还能把结果投递回聊天渠道或 webhook。Standing Orders 则是把长期操作规则写进工作区文件里，规定作用范围、触发条件、审批门槛和升级规则。三者连起来，才像一个真正“值班”的合作者。

第四种岗位，

是“分工器”。

OpenClaw 支持多 agent 路由，每个 agent 可以有自己的 workspace、agentDir、sessions 和 auth profiles；也支持 sub-agents，在独立 session 里做后台工作，完成后再把结果汇报回来。这个设计非常适合把“写代码”“内容整理”“社交发布”“私人事务”拆成不同 agent，而不是全塞给一个混合人格的大虾。
