Skills

skills 是策略面，负责把一类重复任务整理成可复用的做法。
skills 这个词并不是整个行业里完全统一的标准词，但至少在 Claude 官方体系里，它被明确定义为一组可复用的指令、支持文件和可选执行逻辑，适合把反复出现的 playbook、checklist、多步流程打包起来；而且 skill 的内容是按需加载，不必像全局上下文那样每次都吃 token。Claude 官方还区分了 built-in commands 和 bundled skills，前者更像固定逻辑，后者更像 prompt-based playbook。
skills 最像“作业手册”。不是直接执行，而是告诉模型这类任务通常怎么拆、怎么检查、怎么停。
skills 的价值在于把模糊任务变成较稳定的流程，但它仍然依赖模型理解，所以比纯 CLI 更容易漂。
当任务反复出现，但每次材料不同，比如“做一次代码审查”“把研究资料整理成讲稿”“每周巡检某些站点并输出摘要”“按某种标准评价一批结果”，优先 skills。因为这里的核心资产不是某条命令，而是那套拆解方式、检查表、停止条件。Claude 官方文档对 skill 的定位，正是这类可复用 playbook。

MCP

MCP 是连接面，负责让大模型用统一协议去接外部数据、工具和工作流。MCP 本身采用 client-server 架构，主机应用连接一个或多个 MCP server；协议层明确了 tools、resources、prompts 等能力边界
MCP 是开放协议，不是某个单独产品。官方文档把它定义为连接 AI 应用与外部系统的标准方式，能连接数据源、工具和工作流；它有 host、client、server 的分工，服务器可以暴露 resources 和 tools，tool 由模型发现和调用，但规范同时强调安全与 human-in-the-loop。
MCP 最像“标准化插座”。重点不在某个任务，而在于把外部能力稳定接进来，让不同 host 都能以近似一致的方式调用。
MCP 能把外部世界接进来，能力最强，但权限、鉴权、 schema、服务器稳定性、确认机制都变成了新的风险面。MCP 规范本身就明确提醒了授权、安全和人工确认的重要性。
当任务需要让模型稳定接触外部系统，比如 GitHub、Slack、数据库、文件系统、日历、搜索、内部 API，尤其是希望这套连接能被不同客户端复用时，优先 MCP。因为你要解决的不是“一次执行”，而是“能力接入和标准化暴露”。这正是 MCP 的设计中心。

CLI

CLI 是执行面，负责把明确命令打到系统上。
CLI 则是经典命令行界面。它本质上是用文本命令与操作系统或程序交互，优势在于效率、远程操作、脚本化和可复现。也正因为它足够直接，所以特别适合明确、稳定、可参数化的操作。
CLI 最像“扳手”。命令是什么，系统就做什么。模型在这里更像一个会写命令的人。
CLI 的可靠性通常最高，但前提是任务已经足够清晰。它在“不知道该做什么”时并不会更聪明。
当任务已经很明确，比如“批量转码、抓文件、跑测试、部署、重命名、生成缩略图、调用本地脚本”，优先 CLI。因为这里真正稀缺的不是“理解”，而是“精确执行”。CLI 还天然适合脚本化和审计。
